    Online Development (sock)¶

    Maybe you've thought about... playing a game with friends?
    The networking library can make that dream come true.

    How networking works¶

    Before you start development, let’s look at how two computers connect.
    Think about the game MINECRAFT and how it does networking.

    Clearly, when entering a server address, most of the time we see it represented like this:

    123.456.789.012:25565
mc.hypixel.net

    The common point between these addresses is that they use the format [IP address]:[port].

    You may have noticed that mc.hypixel.net appears to have no port, but in fact its port defaults to 25565, so it is equivalent to mc.hypixel.net:25565.

    Putting Minecraft aside, the things needed for networking are:

    • IP address: like the location of a building; it locates the computer running the server.
    • Port number: like the room number inside that building; it tells the computer which program should receive the data.

    So our network address still follows this format. When you want to join someone else’s room later, ask for both the IP and the port number, then add them to join the game.

    Networking challenges¶

    If you really want to create a multiplayer UT-style game (for example, utmv, DFC, etc.), the first challenge you will encounter is latency.

    Because latency exists, the feel of games that require precise positional control (such as FPS) can degrade sharply. There are many remedies for this:

    In public-network games, code-level approaches to fight latency mainly include prediction, interpolation, lag compensation, protocol optimization, and state synchronization. Developers have invented many advanced techniques, such as Quake’s client-side prediction and Valve’s lag compensation, to improve game experience.

    Method Category Specific Technique Description
    Client prediction Input prediction The client responds immediately to local player input while sending the input to the server for validation and synchronization, reducing perceived delay.
    Interpolation Entity interpolation The client smooths the positions of other players or objects based on server snapshots to eliminate stuttering.
    Lag compensation Backward reconciliation The server processes events by rewinding time to match the game state when the client sent input, ensuring fairness.
    Protocol optimization Use UDP instead of TCP Custom UDP protocols (such as reliable UDP) reduce connection overhead and retransmission latency, suitable for real-time games.
    State synchronization Deterministic lockstep All clients run identical logic and only sync inputs, reducing network traffic but requiring high consistency.
    Server architecture Authoritative server The server is the single authority for game state; clients only display approximate results to prevent cheating and synchronization issues.
    Bandwidth optimization Compression and priority Compress network packets and prioritize important updates (such as position updates) to reduce latency impact.

    In detail¶

    • Client prediction: allows the client to predict the result of player actions, such as movement or shooting, then reconcile with the server state. If the server corrects the state, the client smooths the adjustment, which reduces perceived input delay.

    • Interpolation: the client stores entity state snapshots received from the server and interpolates positions during rendering so other players’ movement appears smooth. For example, Source engine interpolation computes intermediate states based on timestamps to prevent teleporting.

    • Lag compensation: when the server receives a client event, it considers network delay and rewinds the game world state for processing. For example, in shooting games, the server calculates the historical position of a player at the moment a shot was fired, ensuring that high-latency players can still hit their targets. Valve’s algorithm is a classic example.

    • Protocol optimization: games often use custom UDP protocols (such as ENet or custom implementations) with reliability mechanisms (like sequence numbers and acknowledgements) to avoid TCP head-of-line blocking. Developers have created snapshot protocols to synchronize state efficiently.

    • State synchronization: approaches include deterministic lockstep (suitable for turn-based or RTS games) and snapshot interpolation (suitable for FPS). Lockstep requires all clients to stay in sync, while snapshot interpolation lets clients interpolate between updates after receiving them.

    • Server architecture: use an authoritative server model where the server validates all critical actions and prevents client cheating. Combining regional servers or edge computing (such as AWS GameLift) can reduce physical latency.

    • Bandwidth optimization: reduce latency by compressing data (such as delta compression that only sends changed parts), using priority queues (send important updates first), and adaptive update rates (adjust send frequency based on network conditions).

    What to say¶

    Of course, I am glad someone would use SoulEngine to develop an FPS-style game, but I feel it is a bit unfair to the developers. This engine is best suited for lightweight RPGs.

    Example demonstration¶

    You may know this library exists, but still not be sure how to use it to make an online game.

    No problem. I will give an example here, but note that it does not include any data encryption. To prevent cheating, the developer must implement encryption themselves.

    if (love.system.getOS() == "Windows") then
        local handle = io.popen("chcp 65001", "r")
        handle:close()
    end

local sock = require "sock"

function love.load()
    local PORT = 51234
    local isServer = false
    local client = nil
    local server = nil

    -- 1. First try to create a server
    local serverSuccess, serverObj = pcall(sock.newServer, "localhost", PORT)
    if serverSuccess then
        isServer = true
        server = serverObj
        print("[Startup Mode] Server mode (port: " .. PORT .. ")")

        -- Server stores all client states in a table
        server.allClientsState = {}

        server:on("connect", function(data, clientPeer)
            local clientId = clientPeer:getConnectId()
            print("[Server] Client #" .. clientId .. " connected.")

            -- Generate a random color and assign an ID for the new client
            local newClientInfo = {
                id = clientId,
                color = { love.math.random(), love.math.random(), love.math.random(), 1 },
                x = 400,
                y = 300
            }
            -- Store it in the server state table
            server.allClientsState[clientId] = newClientInfo

            -- Send a welcome message with assigned info
            clientPeer:send("welcome", {
                yourId = clientId,
                yourColor = newClientInfo.color
            })

            -- Immediately broadcast the current global state to everyone (including the new client)
            broadcastStateToAll()
        end)

        server:on("disconnect", function(data, clientPeer)
            local clientId = clientPeer:getConnectId()
            print("[Server] Client #" .. clientId .. " disconnected.")
            -- Remove from the state table
            server.allClientsState[clientId] = nil
            -- Broadcast the updated state
            broadcastStateToAll()
        end)

        -- Server listens for client state updates
        server:on("clientUpdate", function(data, clientPeer)
            local clientId = clientPeer:getConnectId()
            if server.allClientsState[clientId] then
                -- Update the client's position
                server.allClientsState[clientId].x = data.x
                server.allClientsState[clientId].y = data.y
                -- Immediately broadcast the global state to everyone
                broadcastStateToAll()
            end
        end)

        -- Helper function to broadcast global state
        function broadcastStateToAll()
            -- Send a copy of the state to avoid reference issues
            local stateCopy = {}
            for id, info in pairs(server.allClientsState) do
                stateCopy[id] = { x = info.x, y = info.y, color = info.color }
            end
            server:sendToAll("worldState", stateCopy)
        end

    else
        -- Server creation failed, switch to client mode
        isServer = false
        print("[Startup Mode] Client mode (connecting to server...)")
        client = sock.newClient("localhost", PORT)

        -- Client local state
        client.myId = nil
        client.myColor = nil
        client.myImage = nil
        client.mouseX, client.mouseY = 400, 300

        -- Store the global state received from the server
        client.worldState = {}
        -- Timer to control state update frequency
        client.stateUpdateTimer = 0
        client.stateUpdateInterval = 0 -- send each frame by default, but this can be adjusted to 0.05 seconds if needed

        client:on("connect", function(data)
            print("[Client] Connected to server.")
            -- Load the image after connection
            client.myImage = love.graphics.newImage("image.png")
        end)

        client:on("welcome", function(data)
            print("[Client] Received assigned ID: " .. data.yourId)
            client.myId = data.yourId
            client.myColor = data.yourColor
        end)

        client:on("worldState", function(data)
            -- Update the local copy of the global state received from the server
            client.worldState = data
        end)

        client:on("disconnect", function(data)
            print("[Client] Disconnected from server.")
        end)

        client:connect()
    end

    G_IS_SERVER = isServer
    G_CLIENT = client
    G_SERVER = server
end

function love.update(dt)
    if G_IS_SERVER and G_SERVER then
        G_SERVER:update()
    elseif G_CLIENT then
        G_CLIENT:update()
        -- Update local mouse position
        G_CLIENT.mouseX, G_CLIENT.mouseY = love.mouse.getPosition()

        -- Periodically send local position updates to the server
        if G_CLIENT.myId then -- ensure we have an ID
            G_CLIENT.stateUpdateTimer = G_CLIENT.stateUpdateTimer + dt
            if G_CLIENT.stateUpdateTimer >= G_CLIENT.stateUpdateInterval then
                G_CLIENT.stateUpdateTimer = 0
                G_CLIENT:send("clientUpdate", { x = G_CLIENT.mouseX, y = G_CLIENT.mouseY })
            end
        end
    end
end

function love.draw()
    if G_IS_SERVER then
        love.graphics.setColor(1, 1, 1)
        love.graphics.print("Server running (port: 51234)", 10, 10)
        love.graphics.print("Client count: " .. (G_SERVER and tostring(G_SERVER:getClientCount()) or "0"), 10, 30)
        love.graphics.print("Syncing all client positions in real time...", 10, 50)

        -- Optional server visualization of all client positions
        for id, info in pairs(G_SERVER.allClientsState or {}) do
            love.graphics.setColor(info.color[1], info.color[2], info.color[3])
            love.graphics.circle("fill", info.x, info.y, 20)
            love.graphics.setColor(1,1,1)
            love.graphics.print("Client#"..id, info.x - 20, info.y - 40)
        end
    elseif G_CLIENT then
        -- Client view: draw all client images based on the global state
        if G_CLIENT.myImage and G_CLIENT.worldState then
            local img = G_CLIENT.myImage
            local w, h = img:getDimensions()

            -- Draw all clients received from the server
            for clientId, clientInfo in pairs(G_CLIENT.worldState) do
                love.graphics.setColor(clientInfo.color[1], clientInfo.color[2], clientInfo.color[3], 1)
                love.graphics.draw(img, clientInfo.x, clientInfo.y, 0, 1, 1, w/2, h/2)

                -- Optionally display the ID next to the image
                love.graphics.setColor(1, 1, 1)
                love.graphics.print("#" .. clientId, clientInfo.x - 10, clientInfo.y + h/2 + 5)
            end
        end

        -- Display local client info
        love.graphics.setColor(1, 1, 1)
        love.graphics.print("Client (connected)", 10, 10)
        if G_CLIENT.myId then
            love.graphics.print("My ID: " .. G_CLIENT.myId, 10, 30)
        end
    end
end

function love.quit()
    if G_CLIENT and G_CLIENT:isConnected() then
        G_CLIENT:disconnect()
    end
end

    Architecture and design analysis¶

    This example implements a simple P2P position synchronization demo. Its core design is:

    1. Dynamic role determination: At startup, the program first tries to create a server on the specified port. If successful (port is free), it becomes the server. If it fails (usually because a server is already running on that port), it automatically switches to client mode and connects to localhost.
    2. Authoritative server: The server is the only authority for all game state. It is responsible for:
      • Assigning a unique ID and color to each connected client.
      • Maintaining a global table of all client states (allClientsState).
      • Receiving client position updates and immediately broadcasting the consolidated global state to all clients.
    3. Client simulation and synchronization: The client is responsible for:
      • Simulating its own character based on local input (here, mouse position).
      • Periodically sending its simulated state (position) to the server for validation and synchronization.
      • Receiving and applying the authoritative global state broadcast by the server, then rendering all characters.

    Core mechanisms explained¶

    1. Event-driven connection handling

      • server:on(eventName, callback) / client:on(eventName, callback) is the core API of sock.lua. It registers listeners for network events like "connect", "disconnect", or custom events like "welcome" and "clientUpdate". The callback runs when the event triggers.
      • Key point: Event listeners must be set before calling client:connect() or before the connection begins, otherwise you may miss events.

    2. State synchronization flow

      • Client -> Server: The client sends a custom event with client:send("clientUpdate", {x, y}).
      • Server processing and broadcast: The server updates its state in the "clientUpdate" listener and calls broadcastStateToAll to send the new global state to everyone.
      • Server -> All clients: All clients receive the update in the "worldState" listener and update their local worldState table for rendering.
      • Why send a state copy? The broadcastStateToAll function creates stateCopy to avoid sending a direct reference to allClientsState. This prevents unpredictable bugs if clients accidentally modify received data. It is a defensive programming practice.

    3. Resource allocation timing

      • Note that the client does not load image.png immediately in love.load. The image is loaded in the "connect" callback after the network connection is established (client.myImage = love.graphics.newImage("image.png")). This is generally better logic.

    Practical advice¶

    • Debugging: Use print to log connection, disconnect, send, and receive events. It's the most basic and effective method for debugging network code.
    • Security: This example has no security measures. Any client can send arbitrary data. In a real game, the server must validate all received client data to prevent cheating.
    • Extension: You can easily extend the data sent in clientUpdate and worldState to include more state, such as velocity, animation frame, health, etc.

    By understanding these key points, developers can clearly grasp this example's workflow and build more stable and complex networked game features on top of it.

    Encrypting data¶

    As mentioned above, you may worry about cheating. The library includes binser.lua to help with serialization operations.

    But note that binser.lua is only a serialization/deserialization library. It does not provide any encryption by itself. You must add your own encryption layer.

    -- The library includes sock and binser; make sure to require the correct path. By default they are in the Libraries/Network folder.
local binser = require 'binser'
local sock   = require 'sock'

-- Server side
function serverFunc()
    local server = sock.newServer('*', 22122)
    server:on('connect', function(data, client)
        print('Client connected:', client)
    end)
    server:on('receive', function(data, client)
        -- 4. Deserialize the received data
        local receivedData = binser.deserialize(data)
        print('Received data:', receivedData[1], receivedData[2])

        -- Prepare a response
        local response = {"Server response", 2026}
        -- 1. Serialize the data to send
        local serializedResponse = binser.serialize(response)
        -- 2. Send the serialized string
        client:send(serializedResponse)
    end)
    server:listen()
end

-- Client side
function clientFunc()
    local client = sock.newClient('localhost', 22122)
    client:on('connect', function(data)
        print('Connected to server')
        -- Prepare data to send
        local playerData = { name = "Player1", score = 100, inventory = {"Sword", "Shield"} }
        -- 1. Serialize the data
        local serializedData = binser.serialize(playerData)
        -- 2. Send the serialized string
        client:send(serializedData)
    end)
    client:on('receive', function(data)
        -- 4. Deserialize the received data
        local response = binser.deserialize(data)
        print('Server response:', response[1], response[2])
    end)
    client:connect()
end

    Cheat Engine scanning¶

    We must call in our old friend, Cheat Engine.

    If you don't know what Cheat Engine is, let's briefly understand its numerical strength and concept attack.

    Cheat Engine's numerical strength and conceptual properties¶

    1. Powerful memory modification: CE can directly scan and modify game process memory, enabling cheats such as infinite health, infinite ammo, and one-shot kills. It can even be packaged as a standalone trainer program for distribution.

    2. Low cheating barrier and community support: CE provides a graphical interface and Lua scripting support, allowing even non-programmers to use it quickly. Its cheat table (.CT) sharing mechanism forms a large cheating community that accelerates the spread of tools.

    3. Forces developers into costly offense and defense: To counter CE, developers must deploy complex anti-cheat systems (such as Easy Anti-Cheat or BattlEye). These systems need to monitor abnormal memory access, provide driver-level protection, and enforce server-side validation, creating an ongoing and expensive arms race.

    4. Triggers multi-layer anti-cheat detection: Typical CE operations (such as frequent ReadProcessMemory/WriteProcessMemory calls and debugger attachment) are easy for anti-cheat systems to detect through behavior analysis and signature matching, which can lead to bans and user complaints.

    5. Creates high-dimensional cheat threats: CE is not only used for local single-player games; it can also attack mobile games running in PC emulators. This “dimensional reduction strike” increases the difficulty of detection and defense.

    6. Adds extra technical burden: Developers need to invest extra time in data encryption, dynamic memory addresses, multi-level pointers, and authoritative server validation to make CE modification harder, which increases development cost and maintenance complexity.

    According to game security data, modifier attacks (represented by CE) once accounted for about 11% of game security risk, making it a threat developers must address.

    Can CE be prevented?¶

    For the prevention we mentioned earlier — using binser.lua plus some encryption — it can stop some basic CE exploration.

    For example, if you're a CE beginner, you may give up after facing it.

    Why does this help? The reason is:

    Raw Lua tables, numbers, and other data are stored in memory in relatively regular structures, which are easy for CE to locate through scans like "unknown initial value" -> "value changed". After serialization with binser, those values become a compact binary string before sending. In memory, the original data structure disappears and is replaced by a sequence of bytes that is not easy to interpret directly. This effectively interferes with searches based on value or structural patterns.

    But unfortunately, you cannot prevent all cheating. CE is strong enough that 99.9% of developers cannot completely stop it.

    But if you think about it, developing an online game to play with friends should be about having fun.
