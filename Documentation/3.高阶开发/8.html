    联机开发（sock）¶

    也许你想过......和朋友一起玩游戏？
    联机库可以实现你的梦想。

    联机的原理¶

    在制作之前，我们先来看看两个电脑是如何连接起来的。
    让我们先来想想 MINECRAFT 这款游戏，它是怎么联机的。

    很明显，在输入联机地址的时候，大部分时候，我们都会看到一个地址按如下的方式进行表示：

    123.456.789.012:25565
mc.hypixel.net

    那么这两个地址的共同点就是它们都是以 [IP地址]:[端口] 的形式存在。

    也许你已经发现了 mc.hypixel.net 好像是没有端口的样子，但是实际上它的端口默认为 25565 ，也就是说可以等效于 mc.hypixel.net:25565 。

    抛开MC不谈，我们来总结一下联机所需的内容：

    • IP地址：就像是一栋大楼的位置，它定位了服务器所在的计算机。
    • 端口号：就像是这个大楼里面的房间号，告诉计算机数据应该给哪个程序去用。

    所以在这里，我们的联机地址 仍旧遵循这种格式 。以后当你想要加入别人的房间的时候，请问全 IP 和 端口号 ，再进行添加进入游戏。

    联机的挑战¶

    如果你真的想要创作一个多人UT联机游戏（例如utmv、DFC等），那么首先卡住你的就是延迟问题。

    由于延迟的存在，让很多需要卡精细位置的（比如FPS类）游戏手感直线下降，所以对此，也有很多方法进行补救：

    在公网联机游戏中，对抗延迟的代码层面方法主要包括预测、插值、延迟补偿、协议优化和状态同步等。前人们开发了许多高招，如Quake的客户端预测和Valve的延迟补偿技术，以提升游戏体验。

    方法类别 具体技术 描述
    客户端预测 输入预测 客户端立即响应本地玩家输入，同时将输入发送给服务器验证和同步，减少感知延迟。
    插值 实体插值 客户端对其他玩家或物体的位置进行平滑插值，基于服务器快照消除卡顿。
    延迟补偿 回溯补偿 服务器在处理射击等事件时，回溯时间以匹配客户端发送时的游戏状态，确保公平性。
    协议优化 使用UDP而非TCP 采用自定义UDP协议（如可靠UDP）减少连接开销和重传延迟，适合实时游戏。
    状态同步 确定性锁步 所有客户端运行相同逻辑，仅同步输入，减少网络数据量，但要求高一致性。
    服务器架构 权威服务器 服务器作为游戏状态唯一权威，客户端仅显示近似状态，防止作弊和同步问题。
    带宽优化 数据压缩和优先级 压缩网络数据包，并为关键数据（如位置更新）设置高优先级，减少延迟影响。

    详细解释¶

    • 客户端预测：允许客户端预测玩家动作的结果，如移动或射击，然后与服务器状态调和。如果服务器纠正，客户端会平滑调整，这减少了输入延迟感。

    • 插值：客户端存储服务器发送的实体状态快照，并在渲染时插值位置，使其他玩家的移动看起来平滑。例如，Source引擎的插值技术基于时间戳计算中间状态，避免跳跃。

    • 延迟补偿：服务器在收到客户端事件时，考虑网络延迟，回溯游戏世界状态进行处理。例如，在射击游戏中，服务器会计算子弹命中时玩家的历史位置，确保高延迟玩家也能命中目标。Valve的算法是经典例子。

    • 协议优化：游戏常使用自定义UDP协议（如ENET或自定义实现），结合可靠性机制（如序列号和确认），以减少TCP的队头阻塞问题。前人们开发了如"快照协议”来高效同步状态。

    • 状态同步：方法包括确定性锁步（适合回合制或RTS）和快照插值（适合FPS）。锁步同步要求所有客户端帧率一致，而快照插值允许客户端在收到更新后插值。

    • 服务器架构：采用权威服务器模型，服务器验证所有关键操作，防止客户端作弊。结合区域服务器或边缘计算（如AWS GameLift）可以减少物理延迟。

    • 带宽优化：通过数据压缩（如Delta压缩只发送变化部分）、优先级队列（重要更新优先发送）和自适应更新率（根据网络状况调整发送频率）来减少延迟。

    要说的话¶

    当然，用上了SoulEngine我很欣慰有人会用此开发FPS类游戏，但是我感觉这实在是有点委屈你们开发者了，这个引擎用来开发轻量RPG是最好的选择。

    具体实例展示¶

    可能就算知道了有这个库，但是还是不清楚应该如何使用这个库来制作联机游戏吧？

    没关系，这里我会给出一个示例，不过需要注意的是， 这里面没有写任何数据加密的内容 。为了防止作弊，开发者必须自行写一个数据加密。

    if (love.system.getOS() == "Windows") then
        local handle = io.popen("chcp 65001", "r")
        handle:close()
    end

local sock = require "sock"

function love.load()
    local PORT = 51234
    local isServer = false
    local client = nil
    local server = nil

    -- 1. 首先尝试创建服务器
    local serverSuccess, serverObj = pcall(sock.newServer, "localhost", PORT)
    if serverSuccess then
        isServer = true
        server = serverObj
        print("[启动模式] 服务器模式 (端口: " .. PORT .. ")")

        -- 服务器用于存储所有客户端状态的表格
        server.allClientsState = {}

        server:on("connect", function(data, clientPeer)
            local clientId = clientPeer:getConnectId()
            print("[服务器] 客户端 #" .. clientId .. " 已连接。")

            -- 为新客户端生成随机颜色并分配ID
            local newClientInfo = {
                id = clientId,
                color = { love.math.random(), love.math.random(), love.math.random(), 1 },
                x = 400,
                y = 300
            }
            -- 存入服务器状态表
            server.allClientsState[clientId] = newClientInfo

            -- 发送欢迎消息，包含为其分配的信息
            clientPeer:send("welcome", {
                yourId = clientId,
                yourColor = newClientInfo.color
            })

            -- 立即将当前完整的全局状态发送给所有人（包括新客户端）
            broadcastStateToAll()
        end)

        server:on("disconnect", function(data, clientPeer)
            local clientId = clientPeer:getConnectId()
            print("[服务器] 客户端 #" .. clientId .. " 已断开连接。")
            -- 从状态表中移除
            server.allClientsState[clientId] = nil
            -- 广播更新后的状态
            broadcastStateToAll()
        end)

        -- 服务器监听客户端发来的状态更新
        server:on("clientUpdate", function(data, clientPeer)
            local clientId = clientPeer:getConnectId()
            if server.allClientsState[clientId] then
                -- 更新该客户端的位置
                server.allClientsState[clientId].x = data.x
                server.allClientsState[clientId].y = data.y
                -- 立即广播全局状态给所有人
                broadcastStateToAll()
            end
        end)

        -- 广播全局状态的辅助函数
        function broadcastStateToAll()
            -- 发送状态副本，避免引用问题
            local stateCopy = {}
            for id, info in pairs(server.allClientsState) do
                stateCopy[id] = { x = info.x, y = info.y, color = info.color }
            end
            server:sendToAll("worldState", stateCopy)
        end

    else
        -- 创建服务器失败，转为客户端模式
        isServer = false
        print("[启动模式] 客户端模式 (正在连接服务器...)")
        client = sock.newClient("localhost", PORT)

        -- 客户端本地状态
        client.myId = nil
        client.myColor = nil
        client.myImage = nil
        client.mouseX, client.mouseY = 400, 300

        -- 用于存储从服务器接收的全球所有客户端状态
        client.worldState = {}
        -- 控制状态发送频率的计时器
        client.stateUpdateTimer = 0
        client.stateUpdateInterval = 0 -- 每帧发送一次，但是压力太大，建议调整成0.05秒也可以

        client:on("connect", function(data)
            print("[客户端] 成功连接到服务器。")
            -- 加载图片（此处不分配颜色，颜色由服务器分配）
            client.myImage = love.graphics.newImage("image.png")
        end)

        client:on("welcome", function(data)
            print("[客户端] 收到服务器分配的ID: " .. data.yourId)
            client.myId = data.yourId
            client.myColor = data.yourColor
        end)

        client:on("worldState", function(data)
            -- 收到服务器广播的全球状态，更新本地副本
            client.worldState = data
        end)

        client:on("disconnect", function(data)
            print("[客户端] 与服务器断开连接。")
        end)

        client:connect()
    end

    G_IS_SERVER = isServer
    G_CLIENT = client
    G_SERVER = server
end

function love.update(dt)
    if G_IS_SERVER and G_SERVER then
        G_SERVER:update()
    elseif G_CLIENT then
        G_CLIENT:update()
        -- 更新本地鼠标位置
        G_CLIENT.mouseX, G_CLIENT.mouseY = love.mouse.getPosition()

        -- 定时向服务器发送本机位置更新
        if G_CLIENT.myId then -- 确保已获得ID
            G_CLIENT.stateUpdateTimer = G_CLIENT.stateUpdateTimer + dt
            if G_CLIENT.stateUpdateTimer >= G_CLIENT.stateUpdateInterval then
                G_CLIENT.stateUpdateTimer = 0
                G_CLIENT:send("clientUpdate", { x = G_CLIENT.mouseX, y = G_CLIENT.mouseY })
            end
        end
    end
end

function love.draw()
    if G_IS_SERVER then
        love.graphics.setColor(1, 1, 1)
        love.graphics.print("服务器正在运行 (端口: 51234)", 10, 10)
        love.graphics.print("客户端数量: " .. (G_SERVER and tostring(G_SERVER:getClientCount()) or "0"), 10, 30)
        love.graphics.print("实时同步所有客户端位置中...", 10, 50)

        -- 服务器可视化显示所有客户端位置（可选）
        for id, info in pairs(G_SERVER.allClientsState or {}) do
            love.graphics.setColor(info.color[1], info.color[2], info.color[3])
            love.graphics.circle("fill", info.x, info.y, 20)
            love.graphics.setColor(1,1,1)
            love.graphics.print("Client#"..id, info.x - 20, info.y - 40)
        end
    elseif G_CLIENT then
        -- 客户端窗口：根据全球状态绘制所有客户端的图片
        if G_CLIENT.myImage and G_CLIENT.worldState then
            local img = G_CLIENT.myImage
            local w, h = img:getDimensions()

            -- 遍历从服务器收到的所有客户端状态并绘制
            for clientId, clientInfo in pairs(G_CLIENT.worldState) do
                love.graphics.setColor(clientInfo.color[1], clientInfo.color[2], clientInfo.color[3], 1)
                love.graphics.draw(img, clientInfo.x, clientInfo.y, 0, 1, 1, w/2, h/2)

                -- 在图片旁显示ID（可选）
                love.graphics.setColor(1, 1, 1)
                love.graphics.print("#" .. clientId, clientInfo.x - 10, clientInfo.y + h/2 + 5)
            end
        end

        -- 显示本机信息
        love.graphics.setColor(1, 1, 1)
        love.graphics.print("客户端 (已连接)", 10, 10)
        if G_CLIENT.myId then
            love.graphics.print("我的ID: " .. G_CLIENT.myId, 10, 30)
        end
    end
end

function love.quit()
    if G_CLIENT and G_CLIENT:isConnected() then
        G_CLIENT:disconnect()
    end
end

    架构与设计解析¶

    这个示例实现了一个简单的P2P位置同步演示。其核心设计是：

    1. 动态角色判定：程序启动时，首先尝试在指定端口创建服务器。如果成功（端口未被占用），则成为服务器；如果失败（通常意味着该端口上已有一个服务器在运行），则自动转为客户端并尝试连接 localhost。
    2. 服务器权威：服务器是所有游戏状态的唯一权威来源。它负责：
      • 为每个连接的客户端分配唯一的ID和颜色。
      • 维护一个包含所有客户端状态（位置、颜色）的全局表 (allClientsState)。
      • 接收客户端的位置更新，并立即将整合后的全局状态广播给所有客户端。
    3. 客户端模拟与同步：客户端负责：
      • 根据本地输入（此处为鼠标位置）模拟自身角色。
      • 定时将自身模拟的状态（位置）发送给服务器进行验证和同步。
      • 接收并应用服务器广播的权威全局状态，渲染所有角色。

    核心机制详解¶

    1. 连接与事件驱动

      • server:on(eventName, callback) / client:on(eventName, callback)：这是 sock.lua 的核心。用于注册特定网络事件（如 "connect", "disconnect"）或自定义事件（如 "welcome", "clientUpdate"）的监听器。当事件被触发时，对应的回调函数会被调用。
      • 关键点：事件监听器必须在调用 client:connect() 或开始接收连接之前设置好，以确保不会错过任何事件。

    2. 状态同步流程

      • 客户端 -> 服务器：客户端通过 client:send("clientUpdate", {x, y}) 发送自定义事件。
      • 服务器处理与广播：服务器在 "clientUpdate" 监听器中更新状态，并调用 broadcastStateToAll 函数，通过 server:sendToAll("worldState", stateCopy) 将新状态发送给所有人。
      • 服务器 -> 所有客户端：所有客户端在 "worldState" 监听器中接收并更新本地用于渲染的 worldState 表。
      • 为何要发送状态副本？ 在 broadcastStateToAll 中创建 stateCopy 表是为了避免直接发送 allClientsState 的引用。这可以防止在复杂的后续开发中，因客户端对接收到的数据进行意外修改而引发的不可预测问题，是一种良好的防御性编程实践。

    3. 资源分配时机

      • 注意，客户端在 love.load 中并未立即加载 image.png。图片加载是在 "connect" 事件成功回调后才执行的 (client.myImage = love.graphics.newImage("image.png"))。这确保了网络连接建立后再处理资源，是更合理的逻辑。

    实用建议¶

    • 调试：充分利用 print 输出连接、断开、收发数据等信息，这是调试网络程序最基本有效的方法。
    • 安全：此示例没有任何安全措施。任何客户端都可以发送任意数据。在正式游戏中，服务器必须对所有接收到的客户端数据进行验证（防作弊）。
    • 扩展：可以很容易地扩展 clientUpdate 和 worldState 事件所携带的数据，来同步更多状态，例如速度、动画帧、生命值等。

    通过理解以上要点，开发者可以更清晰地掌握此示例的工作流程，并以此为基础构建更稳定、复杂的网络游戏功能。

    加密数据¶

    正如上面所提到的，你可能很担心如果有人使用手段作弊怎么办，接下来，库里面的 binser.lua 会帮助你进行操作。

    但是要注意的是， binser.lua 只是一个序列化反序列化的库，它自己本身没有任何加密功能，这需要我们自己配一个加密。

    -- 库里面有sock和binser，注意自己加上前面的路径，默认都在Libraries/Network文件夹中
local binser = require 'binser'
local sock   = require 'sock'

-- 服务器端
function serverFunc()
    local server = sock.newServer('*', 22122)
    server:on('connect', function(data, client)
        print('客户端连接：', client)
    end)
    server:on('receive', function(data, client)
        -- 4. 反序列化接收到的数据
        local receivedData = binser.deserialize(data)
        print('收到数据：', receivedData[1], receivedData[2])

        -- 准备回复数据
        local response = {"服务器响应", 2026}
        -- 1. 序列化要发送的数据
        local serializedResponse = binser.serialize(response)
        -- 2. 发送序列化后的字符串
        client:send(serializedResponse)
    end)
    server:listen()
end

-- 客户端端
function clientFunc()
    local client = sock.newClient('localhost', 22122)
    client:on('connect', function(data)
        print('连接到服务器')
        -- 准备要发送的数据
        local playerData = { name = "玩家1", score = 100, inventory = {"剑", "盾"} }
        -- 1. 序列化数据
        local serializedData = binser.serialize(playerData)
        -- 2. 发送序列化后的字符串
        client:send(serializedData)
    end)
    client:on('receive', function(data)
        -- 4. 反序列化接收到的数据
        local response = binser.deserialize(data)
        print('服务器回复：', response[1], response[2])
    end)
    client:connect()
end

    CE的扫描¶

    我们必须请出我们最厉害的老朋友， Cheat Engine 。

    如果你还不知道什么是 Cheat Engine ，我们来快速理解一下它的数值和概念攻击。

    CE的数值强度和概念属性¶

    1. 强大的内存修改能力：CE可以直接扫描和修改游戏进程的内存数据，实现如无限生命、无限弹药、秒杀等作弊效果，甚至能制作成独立的外挂程序（Trainer）进行传播。

    2. 极低的作弊门槛与社区化：CE提供了图形化界面和Lua脚本支持，使得即使不懂编程的用户也能快速上手。其“作弊表”（.CT文件）分享机制，形成了一个庞大的作弊资源社区，加速了外挂的扩散。

    3. 迫使开发者进行高成本攻防：为了对抗CE，开发者必须部署复杂的反作弊系统（如Easy Anti-Cheat、BattlEye）。这些系统需要监控异常内存访问、进行驱动级防护、实施服务器端验证等，这是一场持续且昂贵的“军备竞赛”。

    4. 触发多重反作弊检测：CE的典型操作（如频繁调用ReadProcessMemory/WriteProcessMemory、附加调试器）极易被反作弊系统通过行为分析、特征码比对等方式识别，导致玩家账号被封禁，进而引发用户投诉和流失。

    5. 催生高维度作弊威胁：CE不仅用于修改本地单机游戏，还被用来攻击在PC模拟器上运行的移动端游戏，这种“降维打击”增加了检测和防御的难度。

    6. 增加额外的技术负担：开发者需要为数据加密、动态内存地址、多级指针、服务器权威验证等投入额外开发时间，以增加CE的修改难度，这直接提升了研发成本和维护复杂度。

    据游戏安全数据统计，修改器攻击（以CE为代表）在游戏安全风险中曾占比约11%，是开发者必须重点应对的威胁之一。

    CE能不能改¶

    对于我们之前提到的，使用 binser.lua 和一坨加密代码来实现的防范，是可以阻止CE进行一部分基础探索的。

    比如如果你是个用CE的菜鸟，你可能就此就会放弃修改。

    至于为什么这个有效，原因如下：

    未经处理的Lua表、数字等在内存中以相对规整的结构存放，容易被CE通过 “未知初始值”→“数值变动” 扫描定位。经过 binser 序列化后，这些数据在发送前会变成一段紧凑的二进制字符串。在内存中，原始数据清晰的结构消失了，取而代之的是一串不易直接理解的字节。这能有效干扰基于值特征和结构特征的搜索。

    但是很可惜 ，不可能防范所有作弊现象。CE的数值已经强硬到99.9%的开发者都对他没办法了。

    但是细想想，开发一个联机游戏和朋友玩，自然是以乐趣为主。
